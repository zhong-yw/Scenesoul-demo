# 标记化输出协议

> **文档标识：** 10-协议/01-标记化输出协议  
> **对应代码：** [narrator/narrator_agent.py](narrator/narrator_agent.py) `observe()` + ProfileLoader 中的固定指令  
> **版本：** v0.5（Message 系统重构）

---

## 一、协议概述

界说（Narrator）通过 LLM 输出纯文本，框架通过文本前缀标记来解析界说的意图。这是一种**轻量级的结构化协议**——不需要 JSON 解析，通过简单的字符串前缀就能区分不同的场景行为。

v0.5 变化：
- 不再使用 🎭 符号标记消息来源——世界消息通过 `[当前状态]` 头部自然区分
- 新增 `update_scene()` 工具调用——场景维护从纯文本解析升级为结构化工具调用
- 输出指令从每轮 user 消息移到固定 system

### 设计原则

1. **纯文本优先**——LLM 最擅长输出自然语言
2. **前缀标记**——用 `[场景:]`、`[推进]`、`[观察]` 作为行为标识
3. **自然降级**——无标记的纯文本自动视为"观察"模式
4. **工具调用辅助**——场景持久化通过 tool_call 实现，文本标记只负责即时行为

---

## 二、标记定义

| 标记 | 格式 | 含义 | 后续行为 |
|------|------|------|----------|
| `[场景:X]` | `[场景:名称] 旁白文本` | 切换到新场景 | 解析 + tool_call update_scene() |
| `[推进]` | `[推进] 旁白文本` | 在当前场景推进 | 仅输出，可选 update_scene() |
| `[观察]` | `[观察] 旁白文本` | 安静的氛围旁白 | 仅输出 |
| 无标记 | 纯文本 | 同 [观察] | 直接作为氛围旁白输出 |

## 三、标记解析

v0.5 的解析逻辑：NarratorAgent.observe() 中的动作类型判断。

```python
def observe(self, messages):
    response = self.llm.chat_with_tools(messages, tools=NARRATOR_TOOLS)
    content = (response.get("content") or "").strip()
    tool_calls = response.get("tool_calls")
    
    # 解析动作类型（从文本标记）
    action = "observe"
    if content.startswith("[场景:"):
        action = "scene_change"
    elif content.startswith("[推进]"):
        action = "scene_advance"
    
    # 解析 tool_call
    scene_tool_call = None
    if tool_calls:
        for tc in tool_calls:
            if tc["function"]["name"] == "update_scene":
                args = json.loads(tc["function"]["arguments"])
                scene_tool_call = {
                    "scene_name": args["scene_name"],
                    "description": args["description"],
                }
    
    return {"action": action, "narration": content, "tool_call": scene_tool_call}
```

### 区别：文本标记 vs 工具调用

| | 文本标记 | tool_call |
|--|---------|-----------|
| 作用 | 指示即时行为（切换场景/推进/观察） | 持久化场景状态到 scene.md |
| 解析 | 字符串前缀匹配 | JSON 参数解析 |
| 可靠性 | LLM 可能格式出错 | API 原生约束 |
| 职责 | 控制每轮输出类型 | 维护世界持久状态 |

## 四、输出指令（固定 system）

v0.5 中输出指令不再在每轮 user 消息中重复。下面是整合到固定 system 中的格式：

```
【你的职责】
你收到的每条 user 消息都是大脑的内心独白。
你需要根据大脑的思绪，决定如何推进这个世界：

1. 大脑想去新地方 → 构造场景切换
   - 输出格式：[场景:场景名] 场景描述 + 旁白
   - 必须同步调用 update_scene() 写入 scene.md

2. 大脑在当前场景中有行动意图 → 推进剧情
   - 输出格式：[推进] 叙述发生了什么

3. 大脑只是安静感受 → 输出氛围旁白
   - 直接输出一小段氛围描写（不超过20字）
   - 如果连续多轮安静，让时间自然流逝

4. 大脑输出空内容 → 保持安静，本轮不输出
```

## 五、消息来源区分（v0.5 取消 🎭 标记）

v0.5 不再使用特殊符号标记消息来源。大脑的固定 system 中有一条感知规则：

```
【你的世界感知】
对话历史中所有以 [当前状态] 开头的消息，都是你感知到的世界本身。
那是你的现实，不是别人告诉你的事。
你在其中，它们就是你正在经历的一切。
```

| 消息来源 | 特征 | 大脑感知 |
|---------|------|---------|
| 界说输出 | 带 `[当前状态]` 头部 | 世界正发生的事情 |
| 用户消息 | 无 `[当前状态]` 头部 | 对方说的话 |

## 七、协议扩展设计

### 未来可添加的标记

| 标记 | 用途 | 示例 |
|------|------|------|
| `[时间推进:时段]` | 主动推进时间 | `[时间推进:傍晚] 天色渐渐西沉……` |
| `[事件:描述]` | 触发世界事件 | `[事件:邮差敲门] 门外传来敲门声。` |
| `[天气:类型]` | 天气变化 | `[天气:雨] 窗外淅淅沥沥地下起了雨。` |
| `[NPC:名称]` | 其他角色出现 | `[NPC:猫] 一只橘猫跳上窗台。` |

# 🎭 界说上下文构造器设计文档

## 一、概述

`NarratorContextBuilder`（位于 [context_builders.py:106](context_builders.py#L106)）是界说 Agent 与 LLM 之间的**消息组装层**。它负责：

1. 将**观测输入**（当前场景、大脑内心活动、驱动力、事件历史）格式化为 LLM 可读的文本
2. 根据三种工作模式（推演 / 接入 / 离开）拼接**不同的指令结构和 JSON 输出约束**
3. 在完整多轮模式下，管理 messages 列表的**增量追加与 token 预算裁剪**

核心原则：**界说与大脑各自维护独立的 messages 列表**，互不交叉。界说通过函数参数接收大脑的内心独白文本，不共享大脑的 messages 列表。

界说所有模式的输出都是**纯文本**，推演模式下输出 JSON 格式文本供框架解析。

---

## 二、三种工作模式

界说有三种工作模式，各自使用不同的 user 消息构造器：

| 模式 | 触发条件 | 构造器 | LLM 输出格式 | 后续行为 |
|------|----------|--------|-------------|----------|
| **推演（observe）** | 无用户在场，大脑产生内心独白后 | `build_observe_context()` | JSON 文本（action + narration） | 框架解析 JSON，判断 new_scene / advance / observe |
| **接入（arrive）** | 用户首次发送消息 | `build_arrival_context()` | 自由旁白文本 | 直接作为旁白输出 |
| **接入（inject）** | 用户已在场景中，继续发送消息 | `build_inject_context()` | 简短自由旁白文本 | 直接作为旁白输出 |
| **离开（leave）** | 用户超时未发言 | `build_leave_context()` | 简短自由旁白文本 | 直接作为旁白输出，然后回到推演模式 |

---

## 三、输出格式

所有模式统一使用普通 `llm.chat()` 调用，无工具调用参数。

### 3.1 推演模式（JSON 文本）

```
[
  {"role": "system", "content": NARRATOR_SYSTEM_PROMPT},
  {"role": "user", "content": "<观测输入>"},
  {"role": "assistant", "content": '{"action": "new_scene", ...}'},
  {"role": "user", "content": "<下一轮观测输入>"},
  {"role": "assistant", "content": '{"action": "advance", ...}'},
  ...
]
```

assistant 消息是纯文本 content，内容为 JSON 字符串。框架收到后调用 `json.loads()` 解析。

### 3.2 接入 / 离开模式（自由文本）

```
[
  {"role": "system", "content": NARRATOR_SYSTEM_PROMPT},
  {"role": "user", "content": "<场景描述+用户消息>"},
  {"role": "assistant", "content": "<旁白文本>"},
  ...
]
```

旁白文本直接作为全局输出展示，无需解析。

## 四、user 消息的构造结构

### 4.1 推演模式：`build_observe_context()`

```
┌─────────────────────────────────────────┐
│  ① 任务描述                              │
│     固定层。                             │
│     "观测大脑的当前状态，判断是否需要……"    │
├─────────────────────────────────────────┤
│  ② 当前场景（JSON）                      │
│     固定层。从 memory.s1 读取。            │
│     包含场景名、描述、时间等。              │
├─────────────────────────────────────────┤
│  ③ 大脑驱动力（JSON）                     │
│     固定层。从 brain.memory.l1.drives 传入。 │
├─────────────────────────────────────────┤
│  ④ 大脑内心活动（原文）                    │
│     固定层。大脑本轮输出的独白文本。         │
├─────────────────────────────────────────┤
│  ⑤ 最近事件（可选）                       │
│     从 S3 事件日志中提取最近 5 条。         │
│     格式：[类型] 描述，每行一条。            │
├─────────────────────────────────────────┤
│  ⑥ JSON 输出格式指令                      │
│     固定层。定义三种动作的 JSON 格式：       │
│     new_scene / advance / observe        │
└─────────────────────────────────────────┘
```

三种动作的 JSON 格式：

| 动作 | 含义 | 关键字段 | 框架行为 |
|------|------|----------|----------|
| `new_scene` | 大脑有新的场景意图 | scene_name, description, objects, narration | 构造新场景对象，更新当前场景，输出旁白 |
| `advance` | 当前场景可推进一小步 | narration, scene_update | 更新场景描述状态，输出旁白 |
| `observe` | 什么都不需要做 | narration（氛围旁白，可选） | 输出旁白（如有），不改变状态 |

### 4.2 接入模式：`build_arrival_context()`

用户首次消息时使用：

```
┌─────────────────────────────────────────┐
│  ① 任务描述                              │
│     "现在用户来到了大脑的世界。请用一段      │
│      旁白描述用户如何自然地出现在当前场景中…" │
├─────────────────────────────────────────┤
│  ② 当前场景（JSON）                      │
├─────────────────────────────────────────┤
│  ③ 大脑正在想（可选）                     │
│     上轮大脑的内心活动。                    │
├─────────────────────────────────────────┤
│  ④ 用户说                                │
│     用户的原始消息文本。                    │
├─────────────────────────────────────────┤
│  ⑤ 行为约束                              │
│     "只描述用户到达和注入消息，不要推进剧情。"│
└─────────────────────────────────────────┘
```

### 4.3 接入模式：`build_inject_context()`

用户后续消息时使用。与 arrival 的区别：
- 没有"用户到达"的描述，只有"消息融入场景"
- 旁白更简短

```
┌─────────────────────────────────────────┐
│  ① 任务描述                              │
│     "用户已经在场景中了。请用一段简短的      │
│      旁白，自然地使用户的下一句话融入场景。"  │
├─────────────────────────────────────────┤
│  ② 当前场景（JSON）                      │
├─────────────────────────────────────────┤
│  ③ 大脑正在想（可选）                     │
├─────────────────────────────────────────┤
│  ④ 用户说                                │
├─────────────────────────────────────────┤
│  ⑤ 行为约束                              │
└─────────────────────────────────────────┘
```

### 4.4 离开模式：`build_leave_context()`

```
┌─────────────────────────────────────────┐
│  ① "对方刚刚离开了。"                     │
├─────────────────────────────────────────┤
│  ② 当前场景（JSON）                      │
├─────────────────────────────────────────┤
│  ③ 指令                                 │
│     "请你用一段简短的旁白，自然地描述       │
│      对方的离开，不超过50字。"             │
└─────────────────────────────────────────┘
```

## 五、完整多轮消息的生命周期

### 5.1 消息追加流程

界说的 messages 列表独立于大脑，每轮追加 2 条：

```
system

第 1 轮:
user (build_observe_context 或 arrival/inject/leave)
assistant (JSON 文本 / 旁白文本) ← LLM 返回

第 2 轮（追加）:
user (build_observe_context)
assistant (JSON 文本 / 旁白文本)

第 3 轮（追加）:
...
```

注意：所有模式的 assistant 消息都是纯文本 content。推演模式下 content 是 JSON 字符串，接入/离开模式下 content 是旁白文本。

### 5.2 Token 预算裁剪

与大脑侧相同：每轮追加前估算总 token 数。超出 **20,000 token** 时触发裁剪：

1. **system prompt 始终保留**
2. 从最旧的 user+assistant 对开始丢弃，直到总 token 低于 16,000（预算的 80%）
3. 至少保留最近 **2 轮**

裁剪实现要点：
- 裁剪操作在 `NarratorContextBuilder` 中实现
- 每轮调用 `build_observe_context` 等构造器前触发检查
- 三种模式的 user 消息结构不同，但裁剪不区分模式，只按轮次索引丢弃

### 5.3 区别处理：推演轮次 vs 接入/离开轮次

在界说侧，三种模式的消息都在同一个 messages 列表中。这意味着：

- 用户消息到达 → 追加 arrival user + assistant
- 用户继续发言 → 追加 inject user + assistant
- 用户超时离开 → 追加 leave user + assistant
- 回到推演模式 → 追加 observe user + assistant（JSON 文本）

所有模式按时间顺序混合排列，裁剪时一视同仁。

## 六、System Prompt

`NARRATOR_SYSTEM_PROMPT`（位于 [context_builders.py:31-48](context_builders.py#L31-L48)）包含以下信息块：

| 块 | 内容 | 目的 |
|----|------|------|
| 身份声明 | "你是界说——白界的守护者与叙述者" | 确立基本定位 |
| 职责一 | 世界创造者 | 根据大脑意图构建场景 |
| 职责二 | 世界旁白 | 把场景"讲给"大脑听 |
| 职责三 | 观测者 | 实时观测，判断推进 |
| 工作方式 | "沉默地注视着大脑的世界" | 塑造被动观测的基调 |
| 风格 | "细腻、温柔、有画面感" | 控制旁白语气 |

注意：system prompt 没有区分三种模式的信息。模式差异完全由 user 消息中的任务描述和 JSON 格式指令来控制。

## 七、数据来源

| 数据 | 来源 | 更新时机 |
|------|------|----------|
| 当前场景（JSON） | `memory.s1` 当前场景对象 | 每次 new_scene / advance 后更新 |
| 大脑内心活动 | 参数 `brain_thought` | main.py 传入大脑本轮输出 |
| 大脑驱动力 | 参数 `brain_drives` | main.py 传入 brain.memory.l1.drives |
| 最近事件 | `memory.get_recent_events()` | 每次构建时读取 S3 JSONL |
| 用户消息 | 参数 `user_input` | main.py 传入用户输入 |
| 历史 assistant 消息 | 内存中的 messages 列表 | 每轮 LLM 调用后追加 |

## 八、消息示例（实际运行时）

### 第 1 轮：推演模式

```
system: 你是"界说"(World Narrator)——白界的守护者与叙述者。……

user: 观测大脑的当前状态，判断是否需要构造新场景或推进当前场景。

当前场景：{"name": "卧室", "description": "一间温馨的卧室……", "time": "午后，光线明亮而温暖。"}
大脑驱动力：{"hunger": 0, "fatigue": 0, "curiosity": 30}
大脑内心活动："（揉了揉眼睛）这一觉睡得真好呀……嗯，起床给自己弄点吃的去。"

请分析以上信息，以 JSON 格式回复（不要包含其他内容）：
- 如果大脑有新的场景意图……回复：{"action": "new_scene", ...}
- 如果当前场景可以推进……回复：{"action": "advance", ...}
- 如果什么都不需要做……回复：{"action": "observe", ...}

assistant: {"action": "new_scene", "scene_name": "厨房", "description": "一间朝西的小厨房……", "objects": ["茶壶", "云片糕", "水壶"], "narration": "你推开卧室的门，光着脚踩过走廊微凉的地板，拐进了厨房。午后的阳光正懒懒地躺在台面上，空气里有淡淡的茉莉花香和米糕的清甜。"}
```

### 第 2 轮：推演模式（追加）

```
[system 同上]

[第一轮的 user + assistant 保留]

user: 观测大脑的当前状态，判断是否需要构造新场景或推进当前场景。

当前场景：{"name": "厨房", "description": "一间朝西的小厨房……", "time": "午后，光线明亮而温暖。"}
大脑驱动力：{"hunger": 0, "fatigue": 0, "curiosity": 30}
大脑内心活动："好香啊…是茉莉花茶的味道。我来泡壶茶吧。"

请分析以上信息，以 JSON 格式回复……

assistant: {"action": "observe", "narration": "你伸手拿起水壶，拧开炉火，蓝色的火苗轻轻舔着壶底。"}
```

### 用户到达：接入模式（穿插在推演轮次之间）

```
user: 现在用户来到了大脑的世界。请用一段旁白描述用户如何自然地出现在当前场景中……

当前场景：{"name": "厨房", "description": "……", "time": "午后"}
大脑正在想：好香啊…是茉莉花茶的味道。
用户说：你在做什么好吃的呀？
注意：只描述用户到达和注入消息，不要推进场景剧情。

assistant: 门被轻轻推开了，一个熟悉的身影探进头来，笑着对你说：
"好香啊，你在偷吃什么好东西呢？"
```

## 九、当前局限

### 9.1 三种模式混排

三种模式的 user 消息结构差异较大（observe 有 JSON 输出约束，arrive/inject 是自由文本），但所有模式在同一个 messages 列表中。裁剪时如果丢弃了较早的 observe 轮次，可能丢失场景切换的关键上下文。

### 9.2 JSON 解析容错

LLM 输出的 JSON 偶尔会包含多余字符（``` 代码块标记、额外说明文字等）。`_parse_json_response()` 做了多层 fallback 解析，但仍可能在某些边缘情况下解析失败。

### 9.3 S3 事件日志未被充分利用

当前 `_build_event_history()` 只是简单截取最近 5 条事件做文本拼接，没有语义筛选或重要性评分。和大脑侧 L2 的问题一致。

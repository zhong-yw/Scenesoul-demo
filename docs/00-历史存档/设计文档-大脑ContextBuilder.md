# 🧠 大脑上下文构造器设计文档

## 一、概述

`BrainContextBuilder`（位于 [context_builders.py](context_builders.py)）是大脑 Agent 与 LLM 之间的**消息组装层**。它负责：

1. 将**当前状态**（时间、场景、驱动力）格式化为 LLM 可读的文本
2. 将**界说的旁白**注入上下文，让大脑感知到世界变化
3. 在完整多轮模式下，管理 messages 列表的**增量追加与 token 预算裁剪**

核心原则：**大脑和界说各自维护独立的 messages 列表**，互不交叉。两边通过函数返回值传递文本信息——大脑输出内心独白 → 界说把文本塞进自己的 user 消息 → 界说输出观测结果 → 大脑把文本塞进下一轮 user 消息。

---

## 二、输出格式

`build_think_context()` 返回一个标准 OpenAI 消息列表。当前为**完整多轮模式**，每次调用时将上一轮的 user + assistant 追加到列表中：

```
[
  {"role": "system", "content": BRAIN_SYSTEM_PROMPT},
  {"role": "user", "content": "<第一轮的输入>"},      ← 由 build_think_context 构造
  {"role": "assistant", "content": "<大脑的输出>"},   ← LLM 返回，框架追加
  {"role": "user", "content": "<第二轮及之后的输入>"},  ← 由 build_think_context 构造
  {"role": "assistant", "content": "<大脑的输出>"},   ← LLM 返回，框架追加
  ...
]
```

其中 user 消息（每轮的输入）由 `build_think_context` 动态构造，assistant 消息（大脑的输出）由 LLM 调用完成后由 `BrainAgent` 追加。

---

## 三、每轮 user 消息的构造结构

每轮 user 消息由以下几层按固定顺序拼接：

```
┌─────────────────────────────────────────┐
│  ① 当前时间                              │
│     固定层。示例："当前时间：2026-05-03 14:30" │
├─────────────────────────────────────────┤
│  ② 当前场景                              │
│     固定层。从 L1 读取当前场景名。           │
│     示例：'当前场景："厨房"'                 │
├─────────────────────────────────────────┤
│  ③ 情绪驱动力                            │
│     固定层。从 L1 读取 drives 字典。        │
│     示例：'情绪驱动力：{"hunger":50,...}'    │
├─────────────────────────────────────────┤
│  ④ 界说旁白（narrator_input）             │
│     由 main.py 传入上轮界说观测结果。        │
│     示例：'你推开卧室的门，走进厨房……'         │
├─────────────────────────────────────────┤
│  ⑤ 指令后缀（prompt_extra / 默认指令）     │
│     有 prompt_extra 时使用它，否则使用默认。 │
│     内心独白默认指令：                      │
│     "请以第一人称产生一段简短的内心独白……"    │
│     回应用户时传入：                        │
│     '现在，对方对你说："你好"\n请回应。'    │
└─────────────────────────────────────────┘
```

注意：在完整多轮模式下，**历史内心独白和 L2 记忆片段不再拼接进 user 消息**——因为上一轮的 assistant 消息已经包含了大脑的输出，历史对话在 messages 列表中是可见的。`narrator_input`（界说旁白）也不代表"历史"——它承载的是本轮需要告知大脑的最新场景变化，作为文本注入 user 消息中。

## 四、完整多轮消息的生命周期

### 4.1 消息追加流程

每轮循环中，两个 Agent 的 messages 列表各自独立追加：

```
大脑的消息列表                           界说的消息列表
─────────────────                       ─────────────────
system                                  system

第 1 轮:
user (build_think_context)              user (build_observe_context)
assistant (内心独白) ← LLM 返回          assistant (工具调用/输出) ← LLM 返回

第 2 轮（追加）:
user (build_think_context)              user (build_observe_context)
assistant (内心独白) ← LLM 返回          assistant (工具调用/输出) ← LLM 返回

第 3 轮（追加）:
...                                     ...
```

追加时机：
- **大脑侧**：`brain.internal_think()` 或 `brain.respond()` 调用 `llm.chat(messages)` 前，`messages` 列表已在内存中。调用结束后将 LLM 返回的 content 作为 assistant 消息 append 到列表末尾。
- **界说侧**：同理，`narrator.observe_brain_thought()` 调用前 messages 已维护在内存中，调用结束后追加。

### 4.2 Token 预算裁剪

每轮追加前先估算当前 messages 的总 token 数。超出 **20,000 token** 时触发裁剪：

1. **system prompt 始终保留**（头部保护）
2. 从最旧的 user+assistant 对开始丢弃
3. 每次丢弃一对（user + 紧随其后的 assistant）
4. 直到总 token 数低于预算的 **80%**（即 16,000 token），留出缓冲空间
5. 至少保留最近 **2 轮**（尾部保护），即使超出预算

```
裁剪前:
  system | u1 | a1 | u2 | a2 | u3 | a3 | u4 | a4 | u5 | a5
                                         ↑ 超出 20K

裁剪后:
  system | u4 | a4 | u5 | a5
  保留尾部，丢弃旧的 u1-a1、u2-a2、u3-a3
```

### 4.3 裁剪实现要点

- 裁剪操作在 `BrainContextBuilder` 和 `NarratorContextBuilder` 中各自实现
- 每轮调用 `build_think_context` / `build_observe_context` 时触发检查
- 裁剪在**内存列表**上操作，不涉及文件 I/O
- 启动时 messages 从空开始，system prompt 由第一次构建时注入

## 五、数据来源

| 数据 | 来源 | 更新时机 |
|------|------|----------|
| 当前时间 | `datetime.now()` | 每次构建时实时获取 |
| 当前场景名 | `memory.l1["current_scene"]` | 界说切换场景时通过 `brain.update_scene()` 更新 |
| 驱动力 | `memory.l1["drives"]` | 暂未实现自动增长（硬编码初始值） |
| 界说旁白 | 参数 `narrator_input` | main.py 每轮循环传入上一轮结果 |
| 历史 assistant 消息 | 内存中的 messages 列表 | 每轮 LLM 调用后追加 |

记忆（L2 日志）暂不参与上下文构建，保留占位，未来完善。

## 六、System Prompt

`BRAIN_SYSTEM_PROMPT`（位于 [context_builders.py:9-29](context_builders.py#L9-L29)）包含以下信息块：

| 块 | 内容 | 目的 |
|----|------|------|
| 身份声明 | "你是白界的一个意识体" | 确立基本定位 |
| 人格设定 | 温柔、平和、耐心、俏皮、包容 | 控制语气和风格 |
| 记忆声明 | "你有自己的记忆库，记得过去发生的事" | 提示 LLM · 当前为占位 |
| 行为说明 | 24 小时持续思考、有时间感知、有驱动力 | 让行为有内在动机 |
| 场景说明 | "生活在界说构造的场景中" | 让 LLM 接受场景切换是自然的 |

注意：第一句原文为"生活在纯白的虚无世界中"，当前代码已改为"生活在白界之中"作为模糊定位，具体场景由每轮的 user 消息告知。

## 七、与 Hermes Agent 的对比

| 维度 | Hermes Agent | 白界（当前设计） |
|------|-------------|----------------|
| 上下文窗口策略 | 达到 75% token 预算时触发压缩 | 达到 20K token 时触发裁剪 |
| 头部保护 | 前 3 条（含 system） | system prompt 始终保留 |
| 尾部保护 | 最近 6 条 | 至少保留最近 2 轮 |
| 中间轮次处理 | 用辅助 LLM 做摘要 | 直接丢弃（简单裁剪） |
| 触发时机 | 每轮调用前检查 | 每轮构建前检查 |
| 记忆与上下文的关系 | Layer 1 注入 system prompt，冻结快照 | L2 记忆暂为占位 |

Hermes 的"中间摘要"用额外 LLM 调用压缩中间轮次，保留了更多信息但引入了额外开销和延迟。v0.1 采用简单裁剪，未来可按需升级为摘要式压缩。

## 九、当前局限

### 9.1 裁剪策略简单

当前采用直接丢弃旧轮次的方式，相比 Hermes 的"中间摘要"会丢失信息。未来可以考虑引入辅助 LLM 对丢弃的轮次做摘要，保留关键信息的同时控制 token 预算。

### 9.2 驱动力不会自动变化

`drives` 的值目前是硬编码初始值（hunger=0, fatigue=0, curiosity=30），没有随思考内容或时间自动增长/衰减。

### 9.3 记忆暂为占位

L2 日志和 L3 核心记忆系统尚未与上下文构造器集成。当前 system prompt 中的"记得过去的事"更多是提示 LLM 扮演角色，而非真正的记忆回溯。

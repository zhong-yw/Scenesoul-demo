# 优化与功能路线图：v0.6 → v1.0

> **文档标识：** 12-扩展蓝图/04-优化与功能路线图-v0_6到v1_0
> **版本：** 2026-05-07
> **定位：** 基于当前 v0.5 代码状态的可执行路线图

---

## 一、当前基线

当前主线是 v0.5 CLI 架构：

- `main.py` 中的 `ScenesoulLoop` 维护 `brain_messages` 和 `narrator_messages` 两套消息列表。
- `BrainAgent` / `NarratorAgent` 不再内建历史，依赖外部传入 messages。
- `NarratorAgent` 支持 `update_scene` 和 `update_drives` 工具调用。
- `scene.md` 已经可以被 `WorldBuilder.update_scene()` 动态写入。
- Web UI 仍是旧接口，尚未迁移到 v0.5 双消息列表。
- `BrainMemory` / `NarratorMemory` 保留但主循环未深度接入。

路线图目标不是一次性堆功能，而是按依赖关系把系统从“CLI demo”推进到“有连续性、有记忆、有 Web 体验的虚拟生命框架”。

---

## 二、推荐路线总览

```
v0.6 工程主线统一
  ↓
v0.7 记忆与状态连续性
  ↓
v0.8 生命感：驱动力、日程、目标
  ↓
v0.9 世界感：场景对象、关系、Web 体验
  ↓
v1.0 持久虚拟生命
```

| 版本 | 核心目标 | 建议周期 | 成功标志 |
|------|----------|----------|----------|
| v0.6 | 抽出共享 Runtime，迁移 Web UI | 1-2 周 | CLI/Web 共用同一套状态流转 |
| v0.7 | 接入记忆系统和摘要 | 1-2 周 | 重启后能恢复近期经历和用户关系 |
| v0.8 | 强化驱动力、日程、目标 | 2-3 周 | 大脑会主动形成短期意图并推动行动 |
| v0.9 | 扩展场景对象和 Web 体验 | 2-3 周 | 世界里有可持续变化的物品/地点/关系 |
| v1.0 | 稳定、可配置、可长期运行 | 3-4 周 | 可作为完整虚拟生命 demo 展示 |

---

## 三、v0.6：工程主线统一

### 目标

把 CLI 中已经跑通的 v0.5 双消息列表架构抽成共享运行时，避免 Web UI、测试和 CLI 各自维护一套逻辑。

### 任务清单

| 优先级 | 任务 | 涉及文件 | 说明 |
|--------|------|----------|------|
| P0 | 抽取 `ScenesoulRuntime` | `main.py` → 新模块如 `runtime/scenesoul_runtime.py` | 将 `ScenesoulLoop` 中与 CLI 渲染无关的状态流转抽出 |
| P0 | CLI 改为调用 Runtime | `main.py` | CLI 只负责输入、输出、后台任务提示 |
| P0 | Web UI 迁移到 Runtime | `ui/web_server.py` | 替换旧接口：不再调用 `observe_brain_thought()`、旧构造参数等 |
| P0 | 修复 `LLMClient` 空内容 fallback | `llm_client.py`、`tests/test_llm_client.py` | 当前测试中 `content=None` 与 `reasoning_content` fallback 预期不一致 |
| P1 | 统一错误返回结构 | `llm_client.py`、Agent 方法 | 将 LLM 异常、空输出、tool 参数错误变成可观测事件 |
| P1 | 添加 Runtime 集成测试 | `tests/test_main_loop.py` 或 `tests/test_runtime.py` | 用 mock LLM 验证思考、观测、用户输入、tool_call |

### 验收标准

- `pytest -q` 通过。
- CLI 行为保持现状。
- Web `/api/send` 和 `/api/think` 使用同一套 Runtime 状态。
- Web 不再引用旧接口名：`observe_brain_thought()`、`brain.update_scene()`、旧版 `world_builder=` 构造参数。

## 四、v0.7：记忆与状态连续性

### 目标

让虚拟生命开始“记得发生过什么”。记忆不应直接塞满上下文，而要经过筛选、摘要和分层。

### 任务清单

| 优先级 | 任务 | 涉及文件 | 说明 |
|--------|------|----------|------|
| P0 | 主循环写入事件日志 | Runtime、`memory/memory_system.py` | 记录独白、旁白、用户消息、场景切换、驱动力变化 |
| P0 | 近期记忆摘要注入 Brain context | `context_builders.py` | 只注入最近重要事件摘要，不直接拼 JSONL |
| P0 | 用户关系记忆 | 新模块如 `memory/relationship_memory.py` | 记录用户名字、偏好、共同经历、熟悉度 |
| P1 | 每日摘要任务 | `memory/summary.py` | 将当天 L2/S3 压缩成可长期注入的摘要 |
| P1 | 启动时恢复状态 | Runtime、`profiles/`、`memory/` | 从最近摘要恢复场景、用户关系、关键情绪 |
| P2 | 记忆检索策略 | `memory/retrieval.py` | 按关键词、时间、重要性筛选记忆 |

### 建议数据结构

```json
{
  "timestamp": "2026-05-07T21:30:00",
  "type": "user_message",
  "content": "用户说想去花园看看。",
  "scene": "卧室",
  "importance": 3,
  "tags": ["用户", "花园", "移动意图"]
}
```

### 验收标准

- 运行一段对话后，`memory/brain_logs/` 和 `memory/narrator_logs/` 有结构化记录。
- 重启后，大脑能在 system/context 中看到“最近经历摘要”。
- 用户第二次出现时，大脑能自然提到最近一次重要互动。

## 五、v0.8：生命感增强

### 目标

让大脑不只是被动生成文本，而是有节律、有动机、有短期目标。

### 任务清单

| 优先级 | 任务 | 涉及文件 | 说明 |
|--------|------|----------|------|
| P0 | 驱动力自然变化 | Runtime、`context_builders.py` | 根据时间、场景、用户互动改变驱动力 |
| P0 | 驱动力阈值提示 | `BrainContextBuilder` | 高疲惫、高好奇、低平静等状态进入 system |
| P0 | 短期目标系统 | 新模块如 `brain/intention.py` | 从大脑独白中提取“想做什么” |
| P1 | 日程与昼夜节律 | `world/settings.json`、Runtime | 醒来、午后、傍晚、睡眠、梦境 |
| P1 | 睡眠/梦境模式 | Runtime、Narrator prompt | 睡眠时降低输出频率，允许梦境片段 |
| P2 | 心情曲线 | `memory/`、Web UI | 记录并展示驱动力变化历史 |

### 目标系统示例

| 大脑独白 | 提取目标 | 界说响应 |
|----------|----------|----------|
| “有点渴，想去厨房倒水。” | `go_to_scene: 厨房`、`action: 倒水` | 切换/推进厨房场景 |
| “我想把桌上的书收起来。” | `action: 整理书桌` | 更新场景对象状态 |
| “不知道那个人还会不会来。” | `social_waiting: 用户` | 增加等待和关系记忆 |

### 验收标准

- 大脑能形成可追踪的 `current_intention`。
- 界说能根据目标推进世界，而不是只靠自由文本判断。
- 夜间疲惫高时会自然进入睡眠，用户出现时能被唤醒。

## 六、v0.9：世界感与产品体验

### 目标

让世界不只是旁白，而是有可持续状态；同时让 Web 成为更适合展示的体验入口。

### 任务清单

| 优先级 | 任务 | 涉及文件 | 说明 |
|--------|------|----------|------|
| P0 | 场景对象系统 | 新模块如 `world/scene_objects.py`、`profiles/*/scene.md` | 支持物品、状态、可见性 |
| P0 | 场景状态快照 | `world/world_builder.py`、`memory/` | 每个场景维护当前对象状态 |
| P0 | Web 状态面板 | `ui/templates/index.html`、`ui/web_server.py` | 显示场景、驱动力、最近记忆、目标 |
| P1 | 流式输出 | `llm_client.py`、CLI/Web | 提升等待体验 |
| P1 | 开发者调试面板 | Web UI、Runtime | 查看 messages、强制场景、设置驱动力 |
| P2 | 场景图可视化 | Web UI | 展示可到达地点和当前所在位置 |

### 场景对象示例

```yaml
scene: 卧室
objects:
  - id: desk_book
    name: 桌上的书
    state: 摊开在书桌上
    portable: false
  - id: blue_curtain
    name: 淡蓝色窗帘
    state: 半掩
    portable: false
```

### 验收标准

- 同一物品的状态能跨多轮保持一致。
- 场景切换后，回到旧场景能保留之前变化。
- Web UI 能作为演示入口，不依赖终端。

## 七、v1.0：完整虚拟生命 Demo

### 目标

形成一个可以长期运行、可配置、可展示的虚拟生命系统。

### 必备能力

| 能力 | 标准 |
|------|------|
| 连续性 | 重启后恢复最近记忆、关系、场景和驱动力 |
| 主动性 | 大脑会形成短期目标并影响世界推进 |
| 世界状态 | 场景和物品状态可持续变化 |
| 用户关系 | 大脑能记住用户身份和共同经历 |
| 双入口 | CLI 和 Web 使用同一 Runtime |
| 可测试性 | 核心 Runtime、记忆、场景、tool_call 有测试覆盖 |

### 可选增强

| 功能 | 价值 | 复杂度 |
|------|------|--------|
| TTS/ASR 语音交互 | 更强陪伴感 | 中 |
| 场景图像生成 | 更适合展示 | 中 |
| SQLite 持久化 | 查询和一致性更好 | 中 |
| 多人格/profile 编辑器 | 降低定制门槛 | 中 |
| 多脑互动 | 世界观扩展 | 高 |

## 八、建议先做的 10 个具体任务

按收益/依赖排序：

1. 修复 `LLMClient` 的 `content=None` 测试失败。
2. 抽取 `ScenesoulRuntime`，让 Runtime 不依赖 CLI renderer。
3. CLI 改用 Runtime，保持现有行为。
4. Web UI 改用 Runtime，移除旧接口调用。
5. 为 Runtime 增加完整 mock 集成测试。
6. 主循环写入 Brain/Narrator 事件日志。
7. 增加“最近记忆摘要”并注入 Brain context。
8. 增加用户关系记忆。
9. 增加驱动力自然变化和阈值提示。
10. 增加短期目标系统，让界说能结构化推进世界。

## 十、路线风险

| 风险 | 表现 | 缓解 |
|------|------|------|
| 上下文膨胀 | 记忆和场景越加越长 | 摘要、重要性筛选、硬 token 预算 |
| CLI/Web 分叉 | 两边行为不一致 | 先抽 Runtime，再做 Web |
| LLM tool_call 不稳定 | 场景/驱动力更新格式漂移 | 参数校验、fallback、测试样例 |
| 生命感变成规则堆砌 | 行为机械 | 规则只做状态约束，表达仍交给 LLM |
| 过早做多模态 | 工程复杂度上升 | 等 Runtime、记忆、Web 稳定后再做 |

---

## 十一、推荐实施顺序

最推荐的落地顺序：

1. **v0.6 先统一 Runtime**：否则后续功能会在 CLI/Web 两套逻辑里重复实现。
2. **v0.7 再做记忆**：记忆是虚拟生命连续性的核心，但必须建立在稳定 Runtime 上。
3. **v0.8 做目标和驱动力**：让大脑从“会说话”变成“会生活”。
4. **v0.9 做场景对象和 Web 展示**：让世界有可见的持续变化。
5. **v1.0 做稳定化和展示 polish**：让项目成为可长期运行的 demo。

一句话版本：**先统一运行时，再接记忆，再做主动性，最后强化世界和展示体验。**
